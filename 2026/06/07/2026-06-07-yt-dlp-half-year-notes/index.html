yt-dlp 半年使用记录：从命令行恐惧到每天用

发表于2026-06-07|更新于2026-06-07|技术笔记

事情是这样的

半年前我下载一个教程视频，找了半天在线工具不是弹窗就是恶意重定向。最后实在受不了，在某个论坛看到有人用一行命令行搞定：

当时第一反应是这玩意儿看着就像会炸的。但我还是试了。结果下载完一看，视频质量比网页上看的还好——网页压缩得连字幕都糊了，命令行出来的版本连背景杂音都清晰。

从那以后，yt-dlp 就躺进了我的日常工具箱。

最初的踩坑

第一次用踩了三个坑，都是文档里写了但你不会注意的东西。

坑一：格式选择器。 刚上手的时候用 -f best，下载完发现视频和音频是分离的两个文件。后来查了才知道，在 yt-dlp 里 best 只是一个别名，实际解析出来就是 bestvideo + bestaudio，不会自动合并。正确的姿势是：

这个写法的意思是：优先下载音视频分离的最高质量版本然后合并，如果网站不支持分离就退而求其次用 best。加引号是因为 shell 会把 + 和 / 做特殊处理。

坑二：元数据丢失。 用 ffmpeg 合并的时候，视频的作者、标题、封面这些信息会丢。加上 --embed-metadata 或者 --embed-thumbnail 能补回来。我用 mpv 看视频的时候开了文件列表，没元数据的文件名是一串 hash，根本不知道自己下载的是什么。

**坑三：下载速度。有些网站对单一连接限速严重，yt-dlp 支持分片下载：

四个线程并发下载，速度能翻倍。不过要注意，有些服务器会因为这个判定你为恶意请求直接封 IP。

--audio-quality 0 是最高的 192kbps（ffmpeg 的质量等级，0 对应 192k，5 对应 320k）。这里有个细节：用 ffmpeg -f 的时候质量参数跟直接用 -b:a 不一样，-b:a 192k 和 --audio-quality 0 实际上可能输出不同的码率，因为 ffmpeg 对 AAC 和 MP3 的质量映射不同。我试了半年才发现这个差异。

遇到的一些奇怪场景

YouTube 会员专属视频。 这个不需要特殊操作，yt-dlp 本身只下载公共可访问的内容。如果你需要登录，可以：

但说实话，这种场景我几乎没用到。需要登录才能看的视频，一般也不值得下载下来。

下载速度突然变慢。 有两次遇到 yt-dlp 突然只跑到 100KB/s，查了一下日志，发现是被服务端限速了。加上 --retries 3 --socket-timeout 30 能缓解——重试三次，每次超时三十秒。但这治标不治本，有些平台就是针对 CLI 工具做了降速。

下载完文件无法播放。 有一次下载的视频在 VLC 里打不开，在 mpv 里能播但画面是花的。后来查了一下，是 h.265 (HEVC) 编码的问题——有些平台的视频用了非标准的 HEVC 配置，VLC 的硬解码不支持。加上 --recode-video mp4 强制转码能解决，代价是重新编码耗时比较长，而且会有质量损失。

总结

半年来，yt-dlp 帮我省去了大量翻找在线下载工具的时间。命令行工具的学习曲线确实陡峭——格式选择器、输出模板、元数据这些概念，GUI 工具默认给你处理了，但你不需要理解原理；命令行工具把这些暴露出来，你要么接受默认的不理想状态，要么花时间去搞清楚。

我花了一个下午读 yt-dlp 的 README，基本覆盖了 90% 的使用场景。剩下的 10% 遇到具体问题再查。这个投入产出比，我觉得是值得的。
